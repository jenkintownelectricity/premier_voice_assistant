Solving Choppy Audio & Overlapping Sentences

Problem: Audio Overlap

Multiple TTS requests sent before previous audio finishes, causing sentences to play over each other.


Root Causes:

  • Fire-and-forget TTS calls
  • No queue management
  • First-word streaming conflicts with sentence streaming

Solution: TTS Queue System

Implement a queue-based synthesis system that:

  • Tracks _is_speaking state
  • Waits for current audio to finish before starting next
  • Tracks _current_context_id for coordination
  • Handles barge-in by clearing queue

Problem: Choppy Playback

Audio chunks arriving out of order or with gaps between sentences.


Root Causes:

  • Network jitter
  • Inconsistent chunk sizes
  • Buffer underruns

Solution: Buffer Management

Proper audio buffer handling:

  • Jitter buffer for network variations
  • Consistent chunk sizes from TTS
  • Pre-buffering before playback
  • Seamless sentence transitions

Achieving Low Latency with Slower LLMs (622-800ms → 300-400ms)

The Problem: Using high-quality LLMs like Claude, GPT-4, or Gemini results in 622-800ms latency due to their slower TTFT (200-400ms) compared to Groq's ~40ms. Here's how industry leaders solve this.

Why Slower LLMs Have Higher Latency

Typical Slow LLM Flow

User speaks → STT (50ms) → LLM TTFT (300ms) → Wait for sentence (~200ms) → TTS (50ms)

Total: ~600-800ms

Optimized Flow

User speaks → STT (50ms) → Cache/Parallel → TTS (50ms)

Total: ~300-400ms

6 Techniques to Achieve Low Latency with Any LLM

1. Semantic Caching

~50ms vs 500ms+

Cache responses for common phrases. When user says "hello", respond instantly without hitting LLM.

SEMANTIC_CACHE = { "hello": "Hello! How can I help?", "thank you": "You're welcome!", "goodbye": "Goodbye! Take care!", } # ~50ms response vs ~500ms LLM

2. First-Word Streaming

-150ms perceived

Send first 3-4 words to TTS immediately, don't wait for sentence completion.

# After 3-4 words, send to TTS if word_count >= 4: await tts.synthesize(first_chunk) # User hears audio 150ms earlier

3. Speculative Execution

Advanced

How Vapi does it: Start generating response BEFORE user finishes speaking. Use partial transcript to begin LLM call, cancel if user continues.

# On interim transcript if confidence > 0.8: speculation = llm.start(partial) # Cancel if user keeps talking if user_continues: speculation.cancel()

4. Parallel Processing

-100ms

Run multiple operations concurrently instead of sequentially.

  • Start TTS connection while LLM processes
  • Pre-warm TTS with voice model
  • Stream to TTS as tokens arrive

5. Response Delay Optimization

-300ms

Reduce the artificial delay before responding. Most platforms default to 400-600ms.

Default:400-600ms
Optimized:100-150ms
Savings:300ms+

6. LLM Routing

Smart

Route simple queries to fast LLM (Groq), complex to quality LLM (Claude).

# Classify query complexity if is_simple(query): # greeting, FAQ use_groq() # ~40ms TTFT else: # complex reasoning use_claude() # ~200ms TTFT

The Secret: Perceived vs Actual Latency

Key Insight

Users don't feel actual latency - they feel perceived latency.

If audio starts playing at 300ms, users perceive a 300ms response even if the full response takes 2 seconds to complete. This is why first-word streaming is so powerful.


  • Actual latency: Time until response is complete
  • Perceived latency: Time until user hears first audio
  • Target: Perceived latency < 400ms

Best Practices & Recommendations

1. Use Streaming Everywhere

Stream audio to STT, stream LLM tokens, stream TTS audio. Never wait for complete responses.

2. Sentence-Level TTS Streaming

Send each sentence to TTS as soon as it's detected, don't wait for full LLM response. This reduces perceived latency by 200-400ms.

3. First-Word Streaming

Send first 3-4 words to TTS immediately for ultra-low perceived latency. Requires coordination with sentence streaming to avoid overlap.

4. Semantic Caching

Cache responses for common phrases ("hello", "thank you", "goodbye"). Provides ~50ms response vs ~500ms for LLM.

5. TTS Queue Management

Implement a queue to prevent overlapping audio. Track speaking state and only start new synthesis when previous finishes.

6. Barge-In Support

Allow users to interrupt the assistant. Clear TTS queue immediately and start processing new input.

7. Response Delay Tuning

Industry standard: 100-150ms. Too low causes interruptions, too high feels slow. Make configurable per use case.

8. Use Fastest Components

Deepgram Nova-3 (~30ms), Groq (~40ms TTFT), Cartesia Sonic-3 (~30ms TTFB). The stack matters more than individual optimizations.
