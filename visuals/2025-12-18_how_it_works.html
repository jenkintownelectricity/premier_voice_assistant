How Everything Works

A complete technical guide to the HIVE215 Voice AI Platform - from user speech to AI response in ~200ms.

Introduction

HIVE215 is a real-time voice AI platform that enables natural conversations with AI assistants. When a user speaks, their voice is processed through a pipeline that transcribes speech, generates an intelligent response, and speaks it back - all in under 300 milliseconds.

💡 Key Concept

Unlike chatbots where you type and wait, voice AI requires real-time streaming at every step. You can't wait for the full sentence - you process as audio arrives.

The Core Components

  • LiveKit Cloud - Managed WebRTC infrastructure for real-time audio
  • Railway Worker - Your AI agent that processes voice
  • Deepgram - Speech-to-text in ~30ms
  • LLM Chain - Fast Brain → Groq → Claude for intelligence
  • Cartesia - Text-to-speech in ~30ms
  • Supabase - Database, auth, and storage

Why LiveKit?

LiveKit is a managed WebRTC platform. Here's why we use it instead of raw WebRTC:

Feature Raw WebRTC LiveKit Cloud
Setup complexity Weeks of work 5 minutes
TURN servers You manage Included globally
Scaling Build yourself Automatic
Agent framework Build yourself livekit-agents SDK
Latency ~50-100ms ~10-20ms
✅ Key Point

You don't host LiveKit. LiveKit Cloud is a managed service. Your Railway worker connects to LiveKit Cloud - it doesn't run LiveKit itself.

Voice Flow Explained

Here's exactly what happens when a user speaks:

0ms
User starts speaking
Browser captures audio via WebRTC, streams to LiveKit Cloud
~10ms
LiveKit receives audio
Audio packets arrive at LiveKit's edge servers globally
~20ms
Worker receives audio
Railway worker (subscribed to the room) gets the audio stream
~30ms
Silero VAD detects speech
Voice Activity Detection confirms "user is speaking"
~50ms
Deepgram transcribes
Streaming STT converts audio to text as it arrives
User stops + ~100ms
LLM generates response
Fast Brain/Groq/Claude processes the transcribed text
+ ~30ms
Cartesia synthesizes speech
TTS starts streaming audio back immediately
+ ~20ms
User hears response
Audio streams through LiveKit back to browser

Total: ~200-300ms from user stops speaking to hearing AI response.

Speech-to-Text (Deepgram)

Deepgram's Nova-2 model provides streaming transcription with ~30ms latency.

How Streaming STT Works

# Traditional STT (slow)
audio_complete = record_until_silence()  # Wait for full audio
text = transcribe(audio_complete)        # Then transcribe
# Total: 2-3 seconds

# Streaming STT (fast)
for audio_chunk in audio_stream:
    partial_text = transcribe_chunk(audio_chunk)  # Transcribe as it arrives
    # Each chunk: ~30ms
# Total: Real-time!

Why Deepgram?

  • Speed - Fastest production STT available (~30ms)
  • Accuracy - Nova-2 model optimized for real-time
  • Streaming - True streaming, not chunked batch
  • Cost - $0.0043/minute (very affordable)

LLM Chain

The intelligence layer uses a fallback chain for reliability:

LLM Priority Chain:
1. Fast Brain (Modal)  → Your custom LPU, lowest latency
2. Groq Llama 3.3      → ~50ms, ultra-fast fallback
3. Anthropic Claude    → ~100ms, most capable fallback

Why a Fallback Chain?

  • Reliability - If Fast Brain is cold/down, Groq responds
  • Latency optimization - Always use fastest available
  • Cost control - Fast Brain is cheapest, Claude most expensive
💡 Health Checks

The agent checks /health on Fast Brain before each request. If it fails, instantly falls back to Groq without timeout delay.

Text-to-Speech (Cartesia)

Cartesia's Sonic model provides natural speech synthesis with ~30ms latency.

Why Cartesia?

  • Streaming - Starts speaking before full text is generated
  • Quality - Natural, expressive voices
  • Latency - ~30ms to first audio byte
  • Voice cloning - Custom voices from samples

Streaming TTS Flow

# As LLM generates tokens:
LLM: "Hello"         → TTS starts: "Hel-"
LLM: "Hello, how"    → TTS continues: "-lo, how"
LLM: "Hello, how are"→ TTS continues: " are"
# User hears speech BEFORE LLM finishes generating!

Voice Activity Detection (Silero)

Silero VAD is a neural network that detects when users start/stop speaking.

Why VAD Matters

  • Turn-taking - Know when user finished speaking
  • Interruption handling - Stop AI when user interrupts
  • Silence filtering - Don't process background noise
⚠️ Why the image is 5GB

Silero VAD requires PyTorch (~1.5GB). This is the main reason your Docker image is large. The tradeoff is worth it - Silero provides the most accurate real-time VAD available.

Why the 5GB Docker Image?

The image is large because of ML/AI dependencies. Here's the breakdown:

~1.5 GB

~500 MB

~800 MB

Can We Make It Smaller?

  • Remove Silero - Use server-side VAD (~3GB smaller, but less accurate)
  • Pre-built image - Build once, push to Docker Hub
  • Multi-stage build - Separate build and runtime layers

Build Process (nixpacks)

Railway uses nixpacks to build the Docker image. Here's what happens:

nixpacks.toml

[phases.setup]
nixPkgs = ["python311", "python311Packages.pip", "python311Packages.virtualenv", "ffmpeg", "libopus"]

[phases.install]
cmds = [
    "python -m venv /app/venv",
    "/app/venv/bin/pip install --upgrade pip",
    "/app/venv/bin/pip install --no-cache-dir -r requirements.txt",
    "/app/venv/bin/pip install --no-cache-dir 'livekit-agents[deepgram,cartesia,openai,silero]==1.1.7'"
]

[start]
cmd = "/app/venv/bin/python backend/livekit_worker.py start"

Why the Virtual Environment?

Nix-managed Python has PEP 668 protection - you can't pip install directly. The venv bypasses this:

# This fails (PEP 668):
pip install package
❌ error: externally-managed-environment

# This works:
python -m venv /app/venv
/app/venv/bin/pip install package
✅ Success!

LiveKit Cloud

LiveKit Cloud is a managed WebRTC infrastructure service.

What It Does

  • Room management - Creates/destroys voice chat rooms
  • WebRTC routing - Routes audio between participants
  • Agent dispatch - Notifies your worker when users join
  • Global edge - Low-latency connections worldwide

Connection Flow

1. Frontend calls POST /livekit/rooms
2. Backend creates room in LiveKit Cloud
3. Backend returns token to frontend
4. Frontend connects to LiveKit Cloud with token
5. LiveKit notifies Railway worker "user joined room X"
6. Worker joins room and starts voice pipeline

Latency Optimization

Every millisecond matters in voice AI. Here's how we optimize:

Latency Budget

Component Target Technique
WebRTC ~20ms LiveKit edge servers
STT ~30ms Deepgram streaming
LLM ~50-100ms Groq LPU, streaming tokens
TTS ~30ms Cartesia streaming
Total ~200ms

Key Optimizations

  • Streaming everything - Never wait for complete data
  • Groq over Claude - 50ms vs 100ms for LLM
  • Health checks - Skip unhealthy services instantly
  • Warm containers - Modal keep-alive for Fast Brain

LLM Fallback Chain

The agent tries LLMs in order until one responds:

async def get_llm_response(text):
    # 1. Try Fast Brain (custom LPU)
    if fast_brain.is_healthy():
        try:
            return await fast_brain.generate(text)
        except:
            pass

    # 2. Fallback to Groq (ultra-fast)
    if groq_api_key:
        try:
            return await groq.generate(text)
        except:
            pass

    # 3. Final fallback to Claude
    if anthropic_api_key:
        return await claude.generate(text)

    # No LLM available
    raise NoLLMAvailableError()

Nixpacks & venv Explained

Why we use a virtual environment in the build:

The Problem

# Nix-managed Python has PEP 668 protection
$ pip install fastapi
error: externally-managed-environment
× This environment is externally managed

The Solution

# Create isolated venv, install there
$ python -m venv /app/venv
$ /app/venv/bin/pip install fastapi
✅ Successfully installed

Important: Use venv Paths Everywhere

# Wrong (won't find packages):
python backend/main.py

# Correct (finds venv packages):
/app/venv/bin/python backend/main.py
