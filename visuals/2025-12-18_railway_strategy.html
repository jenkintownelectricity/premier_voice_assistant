Railway Deployment Strategy

Decision guide for fixing the venv/start command issue

🚨 The Problem

Root Cause: Nixpacks auto-detects FastAPI and generates its own start command (uvicorn backend.main:app), completely ignoring the Procfile.

Result: The start command runs without the venv activated, so uvicorn and python-dotenv aren't found.

Evidence from build log:

║ start │ uvicorn backend.main:app --host 0.0.0.0 --port ${PORT:-8000} ║
↑ Auto-generated by nixpacks, NOT from Procfile!

📁 Current File State

Procfile
web: . /app/venv/bin/activate && uvicorn ... worker: . /app/venv/bin/activate && python ...
❌ IGNORED by nixpacks
nixpacks.toml
[phases.install] ✅ Creates venv [variables] PATH includes venv [start] ❌ Removed (hoping Procfile works)
⚠️ Partial - no [start]
railway.toml
[build] builder = "nixpacks" [deploy] restartPolicy only
✅ Used for build config

🔍 Why Nixpacks Ignores Procfile

Nixpacks Start Command Detection Order: ───────────────────────────────────────────────────────────────────── 1. [start] in nixpacks.toml → ❌ We removed it ↓ 2. Procfile detection → ⚠️ SKIPPED because... ↓ 3. Framework auto-detection → ✅ WINS! Nixpacks sees: • uvicorn in requirements.txt • backend/main.py with FastAPI app • Generates: uvicorn backend.main:app THE PROBLEM: Nixpacks Python provider has AGGRESSIVE auto-detection. It generates uvicorn command BEFORE checking Procfile if it detects FastAPI.

🎯 Solution Options

Option A: Start Script with Environment Variable
✅ RECOMMENDED
Create a single start script that checks RAILWAY_SERVICE_NAME (auto-set by Railway) to run the correct command. Add [start] back to nixpacks.toml pointing to this script.

✅ Pros

  • All config in files (no UI changes)
  • Single source of truth
  • Works with Railway's multi-service model
  • Venv activation guaranteed

❌ Cons

  • Extra start.sh file
  • Relies on RAILWAY_SERVICE_NAME env var
scripts/start.sh (new file)
#!/bin/bash
source /app/venv/bin/activate

if [ "$SERVICE_TYPE" = "worker" ]; then
    exec python backend/livekit_worker.py start
else
    exec uvicorn backend.main:app --host 0.0.0.0 --port ${PORT:-8000}
fi
nixpacks.toml (add [start])
[start]
cmd = "bash scripts/start.sh"
Railway UI: Set SERVICE_TYPE env var
Web service:    SERVICE_TYPE=web
Worker service: SERVICE_TYPE=worker
Option B: Custom Dockerfile (Replace Nixpacks)
⚙️ MORE CONTROL
Ditch nixpacks entirely. Create a Dockerfile with full control over the build and start process. Railway will auto-detect and use it.

✅ Pros

  • Complete control over everything
  • No nixpacks quirks
  • Industry standard
  • Can use Procfile properly via honcho/foreman

❌ Cons

  • More complex setup
  • Need to manage base images
  • Longer initial setup time
Dockerfile (new file)
FROM python:3.11-slim

WORKDIR /app
RUN apt-get update && apt-get install -y ffmpeg libopus0

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
RUN pip install 'livekit-agents[deepgram,cartesia,openai,silero]==1.1.7'

COPY . .

# Use environment variable to determine which service to run
CMD ["bash", "scripts/start.sh"]
Option C: Force Nixpacks [start] Per Service
🔧 WORKAROUND
Add [start] back to nixpacks.toml for the web service. Create a separate branch or use Railway's "Root Directory" feature for the worker with a different nixpacks.toml.

✅ Pros

  • Simple config change
  • No new files needed

❌ Cons

  • Requires branch/directory tricks
  • Hard to maintain
  • Not truly "in files"
Option D: Monorepo with Service Directories
📁 RESTRUCTURE
Restructure the repo with separate directories for web and worker, each with their own nixpacks.toml. Use Railway's "Root Directory" feature to point each service to its directory.

✅ Pros

  • Clean separation
  • Each service fully independent
  • Standard monorepo pattern

❌ Cons

  • Major repo restructure
  • Duplicate config files
  • More maintenance overhead

💡 My Recommendation: Option A

Option A (Start Script) is the best balance of:

  • All configuration in version-controlled files
  • No major repo changes
  • Works with Railway's existing multi-service architecture
  • Easy to understand and maintain
  • Quick to implement (~10 minutes)

The only requirement is setting SERVICE_TYPE=web or SERVICE_TYPE=worker as an environment variable in each Railway service. This is a one-time UI change, and after that everything is controlled by files.
