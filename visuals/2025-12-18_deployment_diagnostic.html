What Happens When Railway Deploys

1. Railway detects Dockerfile

Railway auto-detects Dockerfile in repo root and uses it instead of nixpacks

2. Docker build runs

pip install requirements.txt + livekit-agents with extras

3. Container starts with CMD

CMD ["bash", "scripts/start.sh"] - runs the start script

4. start.sh checks SERVICE_TYPE

If worker → python backend/livekit_worker.py | else → uvicorn

5. Railway injects PORT env var

Web services get PORT from Railway. Workers don't need it.

Diagnose Your Issue

Q1: Did the Docker build succeed?
Yes - Build completed
No - Build failed
Build Failure

Check build logs for:
  • pip backtracking → Pin all livekit plugin versions to 1.3.8
  • opentelemetry conflict → Use opentelemetry-api>=1.39.0
  • cannot find module → Check requirements.txt syntax
Continue to Q2...
Q2: Does the container start? (See "Starting Container" in logs)
Yes - Container started
No - Container won't start
Container Startup Failure

Check for:
  • Missing CMD in Dockerfile
  • start.sh not executable or missing
  • Syntax error in start.sh
Continue to Q3...
Q3: What error do you see after "Starting Container"?
Invalid value for '--port'
cannot import 'room_io'
uvicorn: command not found
No error - it's running
PORT Variable Not Expanding

Cause: Shell syntax ${PORT:-8000} not being interpreted

Fix: Update scripts/start.sh to use explicit check:
if [ -z "$PORT" ]; then PORT=8000 fi uvicorn ... --port "$PORT"
LiveKit Version Too Old

Cause: livekit-agents version doesn't have room_io module

Fix: Update to livekit-agents==1.3.8 in both requirements.txt AND Dockerfile
Dependencies Not Installed

Cause: pip install failed silently or wrong Python path

Fix: Check Dockerfile has correct pip install commands
Service is running!

If it's the web service, check the health endpoint.
If it's the worker, check LiveKit Cloud for connected agents.
Q4: Is SERVICE_TYPE set correctly in Railway?
Yes - web or worker
No / Not sure
Missing SERVICE_TYPE

Go to Railway UI → Service → Variables → Add:
  • Web service: SERVICE_TYPE=web
  • Worker service: SERVICE_TYPE=worker
Without this, ALL services will run as web (uvicorn).
SERVICE_TYPE is configured correctly.

Error Pattern Reference

Error: Invalid value for '--port': '${PORT:-8000}' is not a valid integer

What's happening:
The shell is NOT interpreting bash variable syntax. The literal string ${PORT:-8000} is being passed to uvicorn instead of the actual port number.
Why:
Railway's container might be using sh instead of bash, or the script isn't being executed as a shell script properly.
Fix: Use explicit if-check instead of bash default syntax:
if [ -z "$PORT" ]; then PORT=8000 fi exec uvicorn backend.main:app --host 0.0.0.0 --port "$PORT"

Error: cannot import name 'room_io' from 'livekit.agents'

What's happening:
The installed livekit-agents version doesn't have the room_io module.
Why:
room_io was added/restructured in livekit-agents 1.x. Version 1.1.7 doesn't expose it properly. Need 1.3.8+.
Fix: Update requirements.txt AND Dockerfile to use 1.3.8:
# requirements.txt livekit-agents==1.3.8 livekit-plugins-deepgram==1.3.8 # ... all plugins at 1.3.8 # Dockerfile RUN pip install 'livekit-agents[deepgram,cartesia,openai,silero]==1.3.8'

Error: ResolutionImpossible - opentelemetry-api conflict

What's happening:
livekit-agents 1.3.8 requires opentelemetry-api>=1.39.0, but you pinned an older version.
Why:
Previous sessions used opentelemetry-api==1.21.0 which worked with older livekit versions.
Fix: Update requirements.txt:
# Change from: opentelemetry-api==1.21.0 opentelemetry-sdk==1.21.0 # To: opentelemetry-api>=1.39.0 opentelemetry-sdk>=1.39.0

Error: pip backtracking (takes 19+ minutes)

What's happening:
pip is trying many version combinations to find compatible packages.
Why:
LiveKit plugins without pinned versions let pip try every combination.
Fix: Pin ALL livekit plugins to the same version:
livekit-agents==1.3.8 livekit-plugins-deepgram==1.3.8 livekit-plugins-cartesia==1.3.8 livekit-plugins-openai==1.3.8 livekit-plugins-anthropic==1.3.8 livekit-plugins-silero==1.3.8 livekit-plugins-turn-detector==1.3.8

Pre-Deployment Checklist

Click items to mark as checked

Code Changes

  • Dockerfile uses python:3.11-slim
  • Dockerfile installs ffmpeg and libopus0
  • requirements.txt has livekit-agents==1.3.8
  • ALL livekit plugins pinned to 1.3.8
  • opentelemetry-api>=1.39.0 (not ==1.21.0)
  • scripts/start.sh uses explicit PORT check (not ${PORT:-8000})
  • Changes pushed to main branch

Railway Configuration

  • Web service has SERVICE_TYPE=web
  • Worker service has SERVICE_TYPE=worker
  • Worker has LIVEKIT_URL set
  • Worker has LIVEKIT_API_KEY set
  • Worker has LIVEKIT_API_SECRET set
  • Worker has DEEPGRAM_API_KEY set
  • Worker has CARTESIA_API_KEY set
  • Custom Start Command is EMPTY (not overriding)

Verification

  • Fast Brain /health returns status: healthy
  • Web service logs show "Starting Web Server on port..."
  • Worker logs show "Starting LiveKit Worker..."
  • No import errors in worker logs
