This version maintains backwards compatibility with existing Cartesia-only assistants while adding multi-provider support through the new tts_provider field.

Feature Overview

HIVE215 now supports 6 TTS providers with per-assistant configuration. Users can select the optimal provider based on latency requirements, voice quality preferences, and cost considerations.

Technical Architecture

Database Schema

-- Migration 011: Add TTS Provider Selection ALTER TABLE public.va_assistants ADD COLUMN IF NOT EXISTS tts_provider VARCHAR(50) DEFAULT 'cartesia'; -- Supported values: cartesia, elevenlabs, deepgram, openai, playht, rime

Worker Initialization

async def _initialize_tts(self) -> bool: provider = self.config.tts_provider.lower() voice_id = self.config.tts_voice_id if provider == "cartesia": self._tts = cartesia.TTS(model="sonic-english", voice=voice_id) elif provider == "elevenlabs": self._tts = elevenlabs.TTS(voice=voice_id, model="eleven_turbo_v2_5") elif provider == "deepgram": self._tts = deepgram.TTS(model=voice_id) elif provider == "openai": self._tts = openai.TTS(model="tts-1", voice=voice_id) # Fallback chain if primary fails return await self._try_fallback_tts()

Fallback Strategy

If the selected TTS provider fails (missing API key, service down), the worker tries providers in this order:

  1. Cartesia (if API key available)
  2. ElevenLabs (if API key available)
  3. Deepgram (reuses STT API key)
  4. OpenAI (if API key available)

Which Provider Should I Use?

Priority: Lowest Latency?

Need sub-50ms time to first byte for real-time conversation

→ Cartesia (~30ms TTFB) - Industry-leading latency
Priority: Free/Low Cost?

Development, testing, or cost-sensitive deployments

→ Rime AI (~60ms TTFB) - Generous free tier
Priority: Voice Quality?

Premium, expressive voices for customer-facing apps

→ ElevenLabs (~150ms TTFB) - Best voice quality
Priority: Unified Billing?

Same provider for STT and TTS to simplify billing

→ Deepgram Aura (~80ms TTFB) - One API key for both
