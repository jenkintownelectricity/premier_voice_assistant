Voice Provider & Skill Integration

Complete integration of TTS provider selection and Fast Brain skill support

Session Overview

This session completed the end-to-end integration of two major features:

  • TTS Provider Selection - Users can choose between Cartesia, ElevenLabs, Deepgram, or OpenAI for voice synthesis
  • Fast Brain Skill Selection - Each assistant can use a specific Fast Brain skill (receptionist, electrician, plumber, etc.)
Root Cause Identified: The fast_brain_skill field was missing from:
  • Database schema (va_assistants table)
  • Backend API models and endpoints
  • Frontend type definitions

Issues Identified & Fixed

BUG Cartesia voices not loading (100+ voices missing)

Cause: list_voices() assumed API returns {"voices": [...]} but Cartesia returns array directly.

File: backend/cartesia_client.py:715

FIX Handle both response formats

if isinstance(result, list):
    return result  # Direct array
elif isinstance(result, dict):
    return result.get("voices", [])

BUG Skills not being saved to database

Cause: fast_brain_skill missing from CreateAssistantRequest, UpdateAssistantRequest models and endpoint field lists.

FIX Added field to all layers

# Backend models
fast_brain_skill: Optional[str] = "default"

# Create endpoint
"fast_brain_skill": request.fast_brain_skill,

# Update endpoint field list
for field in [..., "fast_brain_skill", ...]:

BUG SQL migration function conflicts

Cause: Multiple function overloads existed; CREATE OR REPLACE couldn't determine which to replace.

FIX Dynamic function drop before create

-- Drop all overloads dynamically
FOR func_sig IN
    SELECT oid::regprocedure::text FROM pg_proc p
    WHERE p.proname = 'va_client_get_assistant'
LOOP
    EXECUTE 'DROP FUNCTION ' || func_sig || ' CASCADE';
END LOOP;
